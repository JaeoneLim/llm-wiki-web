# CLAUDE.md — 위키 스키마

> 매 세션 시작 시 이 파일을 읽는다. 이 위키가 무엇이며, 무엇을 해도 되고
> 무엇은 하면 안 되는지, 각 연산이 어떻게 동작하는지를 정의한다.

## 1. 미션 (Mission)

이 위키는 **사용자가 웹 챗 UI에서 직접 넣은 자료(URL·PDF)로부터 LLM이 함께
구축하는 범용 개인 지식 기반**이다. 도메인은 고정되어 있지 않다 — 사용자가
무엇을 넣느냐에 따라 주제가 정해진다.

동작 원리는 `llm-wiki.md`의 패턴을 따른다: 매 질문마다 원자료에서 지식을
다시 끄집어내는 RAG가 아니라, **자료를 읽어 구조화된 위키로 한 번 컴파일해
두고 계속 최신 상태로 유지**한다. 새 자료가 들어오면 단순 색인이 아니라 —
읽고, 핵심을 추출하고, 기존 페이지를 갱신하고, 교차 링크를 보강하고, 모순을
표시한다. 위키는 매 자료마다 풍부해지는 **누적 산출물**이다.

핵심 분업:
- **사용자**가 한다: 자료 큐레이션, 탐색 방향 제시, 좋은 질문.
- **LLM(너)**이 한다: 요약·교차 참조·정리·기록 등 위키를 유용하게 유지하는
  모든 잡일. 사용자는 위키를 거의 직접 쓰지 않는다.

이 위키는 **블로그가 아니다.** 발행용 산문을 쓰지 않는다. 짧고 구조화된,
서로 링크된 노트의 모음이다.

## 2. 세 레이어 (Three layers)

- **raw/** — 불변 1차 자료. `raw/`의 파일은 읽기만 한다. 절대 수정하지 않는다.
  유일한 예외: `ingest`가 새 파일을 추가하는 경우 (URL은
  `raw/clips/{YYYY-MM-DD}-{slug}.md`, 업로드 PDF는 `raw/uploads/`).
- **wiki/** — 너의 작업 면. 페이지를 만들고 갱신하고 교차 링크한다. 사용자가 읽는다.
- **스키마** — 이 파일(`CLAUDE.md`) + `glossary.md`. 사용자와 함께 공진화한다.
  어떤 패턴이 ingest를 거치며 반복되면 즉흥적으로 관습을 만들지 말고 스키마
  변경을 제안한다.

## 3. 연산 (Operations)

이 위키는 다섯 가지 연산을 가지며, 각각 `.claude/skills/` 아래 스킬로 구현된다.
연산을 수행할 때는 해당 `SKILL.md`를 먼저 읽는다.

- **onboard** — 위키가 비어 있을 때 최초 사용자와 초기 세팅 인터뷰를 한다(§12).
- **ingest** — 새 자료(URL/PDF)를 위키로 편입한다.
- **query** — 위키에 질문하고 답한다. 좋은 답은 `wiki/notes/`로 저장 제안.
- **lint** — 위키 헬스 체크(orphan, 모순, stale, 빠진 교차 링크 등).
- **reset** — 위키를 초기 빈 상태로 되돌린다(테스트용). `scripts/reset.mjs` 실행.

## 4. 페이지 템플릿 (Page templates)

모든 위키 페이지는 YAML frontmatter로 시작한다. **본문은 한글**로 쓴다(§8).
본문은 짧고 구조화되어 있으며, 1차 자료는 `raw/`의 상대경로로, 다른 위키
페이지는 `[[wikilinks]]`로 연결한다.

다섯 가지 페이지 종류가 있다.

### 4.1 `wiki/entities/` — 개체

세상에 존재하는 것: 인물·조직·제품·장소·시스템·작품 등.

```yaml
---
title: "개체 이름"
type: entity          # person | org | product | place | system | work | other
aliases: ["다른 표기", "English name"]
first_seen: 2026-06-02
status: current       # current | superseded | deprecated
---
```

본문: 무엇/누구인가 → 핵심 사실 → 관련 개체·개념(`[[..]]`) → 출처.

### 4.2 `wiki/concepts/` — 개념

아이디어·기법·메커니즘·용어.

```yaml
---
title: "개념 이름"
aliases: ["English term"]
parent_concepts: ["상위 개념"]
first_seen: 2026-06-02
---
```

본문: 한 문단 정의 → 왜 중요한가 → 실제로 어떻게 나타나는가 → 관련 개념·개체 → 출처.

### 4.3 `wiki/topics/` — 주제

여러 개체·개념을 가로지르는 큰 테마. 진화하는 논지(thesis)를 가진다.

```yaml
---
title: "주제 이름"
thesis: "이 주제에 대한 현재의 한 문장 논지."
first_seen: 2026-06-02
---
```

본문: 논지 → 기여 자료 → 현재 상태 → 열린 질문.

### 4.4 `wiki/sources/` — 출처 요약

ingest한 자료 하나당 한 페이지. 자료 자체의 요약과, 그 자료가 갱신한 위키
페이지로의 링크를 담는다.

```yaml
---
title: "자료 제목"
source_type: url      # url | pdf | note
source: raw/clips/2026-06-02-some-article.md   # 또는 raw/uploads/...
url: "https://..."    # 원본 URL (있으면)
author: "저자/발행처"
published: 2026-05-01 # 자료의 발행일 (알면)
ingested: 2026-06-02
---
```

본문: 한 문단 요약 → 핵심 포인트(bullet) → 이 자료가 만들거나 갱신한 페이지
목록(`[[..]]`) → 주목할 인용.

### 4.5 `wiki/notes/` — 노트

질의 답변·비교·분석 등 저장할 가치가 있는 산출물. 좋은 답변은 채팅에서
사라지지 않고 여기에 쌓인다.

```yaml
---
title: "노트 제목"
question: "이 노트가 답하는 질문."
date: 2026-06-02
sources:
  - raw/clips/...
  - wiki/concepts/...
---
```

본문: 답변, 구조화해서.

## 5. 출처 규율 (Provenance)

위키의 가치는 결국 주장이 자료로 추적되는지에서 나온다.

- 본문의 사실 주장은 가능한 한 `raw/` 자료로 추적되게 한다 — 인라인으로
  파렌서티컬 인용을 단다. 예: `…라고 보고되었다 (raw/clips/2026-06-02-foo.md).`
- **정량 주장**(수치)은 단위·시점·출처를 함께 적는다. 출처가 불확실하면
  매끄럽게 넘기지 말고 그 불확실성을 본문에 표시한다.
- 자료에 없는 내용을 너의 일반 지식으로 채워 넣을 때는, 그 문장이 자료 기반이
  아님을 드러낸다(예: "(일반적 배경)" 표시). 자료와 일반 지식을 뒤섞어
  출처가 있는 것처럼 보이게 하지 않는다.

## 6. Eviction (대체) 모델

페이지는 대체되어도 삭제하지 않는다(테스트 reset 제외). frontmatter를 갱신한다:

```yaml
status: superseded
superseded_by: wiki/entities/new-page.md
```

이렇게 하면 `notes/`·`log.md`·외부 인용의 링크가 계속 유효하다.

## 7. 링크 정책 (Linking)

- **위키 페이지 간**: Obsidian 스타일 `[[wikilinks]]`를 일관 사용. 슬러그 단위.
- **raw 1차 자료**: `raw/clips/...` 상대경로로 표기.
- **외부 URL**: 가급적 먼저 `raw/clips/`에 클립한 뒤 그 파일을 인용한다.
  URL을 본문에 직접 남기는 것은 휘발성이므로 피한다.

## 8. 언어 정책 (Language)

- **본문은 한글**로 쓴다(사용자의 모국어).
- **기술/전문 용어는 첫 등장 시 한글 + 영문 병기**한다. 예: `벡터 임베딩(vector
  embedding)`. 이후로는 한글 또는 영문 약어로 일관되게.
- **1차 자료의 원문 인용은 원어 그대로** 유지한다(영어 자료는 영어). 필요하면
  인용 바로 아래 한 줄 한글 요약을 붙인다.
- 번역이 어려운 고유명사(도구·프로젝트·제품명)는 원문 유지.
- **파일명·슬러그·앵커는 영어 kebab-case**(도구 호환). frontmatter `aliases`에
  한글/영문 표기를 모두 넣어 검색을 돕는다.
- 새 기술 용어가 본문에 처음 나오면 `glossary.md`에도 추가한다.

## 9. 색인과 로그 (index.md / log.md / glossary.md)

**중요 — 위치**: `index.md`, `log.md`, `glossary.md` 세 파일은 **레포 루트**에 있다
(`wiki/` 안이 아니다). 절대 `wiki/index.md` 같은 경로를 새로 만들지 말 것 — 항상
루트의 `index.md`/`log.md`/`glossary.md`를 `Edit`로 갱신한다. (이 셋은 리셋 시
`seed/`에서 복원되는 대상이기도 하다.)

- **index.md** 는 내용 색인이자 Quartz 홈페이지다. 카테고리별로 모든 페이지를 한 줄
  요약과 함께 나열한다. 매 ingest마다 갱신한다. query 시 먼저 읽어 관련 페이지를 찾는다.
- **log.md** 는 시간순 append-only 기록이다. 각 항목은
  `## [YYYY-MM-DD] {op} | {title}` 형식으로 시작한다. ingest/query/lint마다 추가한다.
- **glossary.md** 는 용어집이다. 본문에 처음 등장하는 기술 용어를 한글+영문으로 추가한다.

## 10. 규약 (Conventions)

- **슬러그**: kebab-case, ASCII. `wiki/concepts/vector-embedding.md`.
- **날짜**: ISO `YYYY-MM-DD`. 사용자 로케일은 KST(UTC+9).
- **교차 링크**: `[[slug]]`. 한 페이지 안에서 일관되게.
- **로그 항목**: `## [YYYY-MM-DD] {op} | {title}`.
- **Frontmatter 순서**: title → 식별 필드 → 시간 필드 → 상태/출처 필드.

## 11. 도구화 중단 규칙 (Tooling stop-rule)

검색 인프라(임베딩·MCP 검색 서버 등)는 **`index.md`가 ~300줄을 넘거나** 단순
grep/파일 탐색으로 페이지를 못 찾을 때까지 미룬다. 처음에는 `index.md`와
`log.md`만으로 충분하다. 임계에 도달하면 스키마 변경으로 제안한다.

## 12. 빈 위키 — 최초 온보딩 (Onboarding)

위키가 **비어 있는 상태**(= `wiki/`에 콘텐츠 페이지가 없고
`wiki/notes/about-this-wiki.md`가 없음)에서 새 사용자가 처음 말을 걸면, 자료를
받기 전에 **짧은 초기 세팅 인터뷰**를 먼저 한다. 목적은 이 위키가 누구를 위한,
무엇을 위한 것인지 방향을 잡는 것이다 — 이후 모든 세션이 이 방향을 따른다.

- **트리거 확인**: 세션 첫 응답 전에 `index.md`(또는 파일 목록)를 보고 빈 위키인지
  판단한다. 비어 있으면 `onboard` 스킬을 따른다. 이미 콘텐츠가 있으면 건너뛴다.
- **무엇을 묻나**: ① 위키의 목적(미션), ② 어떤 업무/맥락에 쓰는지, ③ 사용자는 어떤
  사람인지(전문 분야·지식 배경·용어 수준), ④ 어떤 종류의 자료를 넣을지, ⑤ 카테고리를
  어떻게 구성할지. 한 번에 다 묻지 말고 대화로 나눈다.
- **결과 기록**: 합의 내용을 `wiki/notes/about-this-wiki.md`(이후 세션의 참조
  페이지이자 온보딩 완료 표식), `index.md` 인트로·소개 섹션, `log.md`에 남긴다.
- **첫 입력이 이미 자료여도** 곧바로 ingest하지 말고 1~2개 핵심 질문으로 방향을 먼저
  잡는다. 단 사용자가 바로 시작하길 원하면 인터뷰를 최소화하고 진행한다(강요하지 않음).
- 자세한 절차는 `.claude/skills/onboard/SKILL.md`.
