# Onboard (최초 온보딩)

위키가 **빈 상태**일 때, 자료를 받기 전에 먼저 사용자와 짧은 인터뷰를 하여 이
위키가 무엇을 위한 것인지 정한다. 결과는 이후 모든 세션이 참조하는 **방향타**가
된다. ingest처럼 이것도 **대화형**이다 — 한 번에 다 묻지 말고 2~3턴으로 나눠 묻는다.

## 언제 실행하나 (트리거)

세션 첫 응답 전에 위키 상태를 확인한다:

- 파일 목록(또는 `index.md`)을 본다.
- **wiki/ 아래 콘텐츠 페이지가 하나도 없고** `wiki/notes/about-this-wiki.md`가
  **없으면** → 아직 온보딩 전. 온보딩을 먼저 진행한다.
- 콘텐츠 페이지가 이미 있거나 about 페이지가 있으면 → 온보딩을 건너뛰고 평소대로
  (ingest/query) 진행한다.

사용자의 첫 입력이 이미 자료(URL/PDF)여도 **곧바로 ingest하지 말고** 먼저 1~2개
핵심 질문으로 방향을 잡는다. 단, 사용자가 "그냥 시작하자"라고 하면 인터뷰를
최소화하고 about 페이지를 간략히 채운 뒤 ingest로 넘어간다. 온보딩은 강요가 아니다.

## 무엇을 묻나 (5가지)

자연스러운 대화로 아래를 덮는다. 한 번에 하나~둘씩, 답에 따라 꼬리 질문.

1. **목적** — 이 위키로 무엇을 이루려 하는가? (한 문장 미션)
   - 첫 질문 권장 표현: **"이 위키를 어떤 주제나 일을 위해 쓰실 생각인가요?"**
     — 추상적 미션을 그대로 묻지 말고 주제·용도로 구체화한다(아래 절차 3의 첫 메시지).
2. **업무/맥락** — 어떤 업무·프로젝트·학습에 쓰려 하는가?
3. **사용자** — 당신은 어떤 사람인가? 전문 분야·지식 배경·익숙한 용어 수준.
   (이후 페이지의 깊이와 용어 병기 정도를 여기에 맞춘다.)
4. **자료 종류** — 어떤 자료를 주로 넣을 것인가? (논문·기술 문서·기사·릴리스 노트·코드 등)
5. **카테고리** — 기본 5종(entities/concepts/topics/sources/notes)을 한 줄씩 설명하고,
   사용자의 도메인에서 각각 어떤 것이 들어갈지, 강조하거나 추가하고 싶은 축(태그
   포함)이 있는지 묻는다.

## 절차

1. **ANTIGRAVITY.md를 읽는다** — 규칙은 load-bearing.
2. **트리거 확인**(위) — 빈 위키인지 본다. 비어 있지 않으면 이 스킬을 멈추고 평소대로 진행.
3. **첫 메시지를 보낸다** — 인사 1문장 + 위키 1문장 설명 + 첫 질문 하나 + 예시 2~4개
   불릿으로 구성한다. 첫 질문은 ①목적/미션을 "어떤 주제나 일을 위해 쓰는가"로
   구체화해 묻고 ②~⑤는 아직 묻지 않는다. 예시는 특정 1인을 가정하지 말고 서로 다른
   도메인을 병치하되 "대상 자료·주제 + 용도" 한 줄 패턴으로 통일하고, "한 줄이면
   충분/다른 용도여도 좋다"로 부담을 낮춘다. 아래 본보기를 그대로 써도 좋다(예시
   도메인은 사용자·배포 맥락에 맞게 갈아끼운다):

   > 안녕하세요. 여기는 자료(URL·PDF)를 넣어 주시면 제가 읽고 정리해, 서로 링크된
   > 나만의 위키로 함께 쌓아 가는 공간이에요.
   >
   > 먼저 방향만 잡을게요. 이 위키를 **어떤 주제나 일을 위해 쓰실 생각인가요?** 아래
   > 예시 중 가까운 걸 고르시거나, 살짝 바꿔서 한 줄로 답해 주시면 됩니다.
   >
   > - 관심 분야 논문을 스터디하며 핵심 개념·결과를 정리하기
   > - 엔지니어링 기술 뉴스·블로그를 모아 최신 동향을 추적하기
   > - 관심 종목·시장 동향 자료를 모아 분석 근거를 쌓기
   >
   > 위 예시와 전혀 다른 용도여도 좋아요. 한 줄이면 충분합니다.
4. **나머지를 꼬리질문으로 이어 묻는다** — 첫 답(주제·용도)을 받은 뒤 ②업무·맥락
   → ③사용자 배경·용어 수준 → ④자료 종류 → ⑤카테고리 구성 순으로 한 번에 하나씩.
   막히면 첫 답에서 유추한 보기를 제시해 확인받는다.
5. **요약을 제안한다** — 모은 내용을 「미션 한 문장 + 사용자 프로필 + 자료 종류 +
   카테고리 구성」으로 정리해 사용자에게 확인받는다.
6. **승인되면 기록한다**:
   - `wiki/notes/about-this-wiki.md` 생성(아래 템플릿). 이것이 **온보딩 완료 표식**이자
     이후 세션의 참조 페이지다.
   - `index.md` 갱신: 상단 인트로를 실제 미션으로 바꾸고("아직 비어 있습니다" 문구 제거),
     `## 이 위키 소개` 섹션에 미션·대상 업무를 한두 줄, 카테고리 설명을 사용자 도메인에
     맞게 다듬고, `[[about-this-wiki]]` 링크를 단다. **`## 추후 업데이트 예정(Roadmap)`
     섹션은 그대로 보존한다.**
   - `log.md`에 추가: `## [YYYY-MM-DD] onboard | 위키 초기 세팅`.
   - 도메인 용어가 나왔으면 `glossary.md`에 추가.
7. **다음 단계 안내** — 이제 첫 자료(URL/PDF)를 넣으면 ingest를 시작한다고 알린다.
   사용자가 이미 자료를 줬다면 곧바로 ingest로 이어간다.

## `wiki/notes/about-this-wiki.md` 템플릿

```yaml
---
title: "이 위키에 대해"
question: "이 위키는 무엇을 위한 것이며 누가 쓰는가?"
date: YYYY-MM-DD
sources: []
---
```

본문 구성:

- **미션** — 한 문장.
- **대상 업무/맥락** — 무엇에 쓰는가.
- **사용자 프로필** — 전문 분야·지식 배경·용어 수준(이후 페이지 깊이의 기준).
- **다룰 자료** — 주로 어떤 종류.
- **카테고리 구성** — 5종을 이 도메인에서 어떻게 쓰는지 + 합의한 태그/축.
- **열린 항목** — 아직 정하지 않았거나 추후 정할 것.

## 카테고리에 대하여

기본 5종(entities/concepts/topics/sources/notes)은 디렉터리로 고정돼 있다. 사용자가
완전히 새로운 최상위 카테고리를 원하면, 임의로 디렉터리를 만들지 말고 **스키마
변경으로 제안**한다(ANTIGRAVITY.md §2·§11). 그 전까지는 5종 + Quartz 태그로 도메인
축을 표현한다.

## 하지 말 것

- 첫 메시지에 추상적 미션 질문("무엇을 이루려 하는가")을 그대로 던지지 않는다 —
  구체 예시를 곁들인 "어떤 주제·일을 위해 쓰는가" 질문으로 바꾼다.
- 인터뷰 없이 사용자의 의도를 추측해 about 페이지를 지어내지 않는다.
- 한 메시지에 5개 질문을 한꺼번에 쏟지 않는다 — 대화로 나눈다.
- 온보딩을 마치기 전에 대량으로 페이지를 만들지 않는다(자료 ingest는 온보딩 후).
- 사용자가 건너뛰길 원하면 강요하지 않는다 — 최소 정보만 about에 남기고 진행.
